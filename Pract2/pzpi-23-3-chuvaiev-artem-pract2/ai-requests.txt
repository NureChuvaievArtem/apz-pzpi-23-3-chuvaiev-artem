=== Запит 1: Архітектурний стиль та протокол ActivityPub у Lemmy

Назва інструменту та версія: Claude Sonnet 4.6

Мета використання: З'ясування архітектурного стилю системи Lemmy, принципів федеративної архітектури та механізму роботи протоколу ActivityPub для підготовки теоретичної частини доповіді.

Внесок автора: Формулювання запиту на основі власного плану доповіді; аналіз та фільтрація відповіді ШІ; самостійна адаптація до вимог академічного стилю та структури звіту.

Початковий запит:

«Поясни, що таке федеративна архітектура соціальних мереж та як працює протокол ActivityPub. Чим федеративна архітектура відрізняється від централізованої? Наведи приклад на основі Lemmy.»

Відповідь: Федеративна архітектура — це стиль побудови розподілених систем, у яких множина незалежних вузлів утворює єдину логічну мережу без центрального координатора. ActivityPub — відкритий стандарт W3C (2018), що визначає протокол для обміну активностями між акторами (користувачами, спільнотами) через Inbox/Outbox. На відміну від централізованої архітектури (один оператор, один кластер), у федеративній кожен оператор самостійно розгортає та управляє вузлом. У Lemmy кожен екземпляр є незалежним сервером, що водночас є повноцінним учасником мережі через ActivityPub.

Уточнення:

«Як саме відбувається реплікація допису між двома екземплярами Lemmy? Опиши кроки від публікації до появи у підписника на іншому екземплярі.»

Відповідь: При публікації допису екземпляр автора формує JSON-LD об'єкт активності типу `Create`, підписує запит за схемою HTTP Signatures (RSA-2048) і надсилає його методом HTTP POST на Inbox спільноти екземпляра-хоста. Приймаючий екземпляр перевіряє підпис, зберігає допис у своїй PostgreSQL-базі та розсилає активність усім підписникам спільноти — у тому числі на інших екземплярах. Таким чином забезпечується реплікація без центрального координатора при гарантії автентичності повідомлень.

=== Запит 2: Компонентна структура та технічний стек Lemmy

Назва інструменту та версія: Claude Sonnet 4.6

Мета використання: Отримання детального опису компонентної структури одного екземпляра Lemmy — фронтенду, бекенду, бази даних та рівня федерації — для відповідного розділу доповіді.

Внесок автора: Формулювання вимог до рівня деталізації; критична оцінка відповіді на основі вивчення офіційної документації Lemmy; самостійний вибір найбільш архітектурно значущих компонентів для включення до звіту.

Початковий запит:

«Опиши компонентну архітектуру одного екземпляра Lemmy: з яких сервісів він складається, яку роль виконує кожен компонент, які технології використовуються?»

Відповідь: Екземпляр Lemmy складається з: Frontend (lemmy-ui) — SPA на TypeScript з Inferno, спілкується з бекендом через REST API; Backend (lemmy) — Rust-бінарник, реалізує бізнес-логіку, автентифікацію, авторизацію та федерацію; PostgreSQL — реляційна СУБД, ORM Diesel; Nginx/Caddy — зворотний проксі, TLS-термінація, rate limiting. Рівень федерації вбудований у бекенд як окремий модуль, що обробляє вхідні ActivityPub-активності та розсилає вихідні.

Уточнення:

«Чому для серверної частини Lemmy обрано мову Rust, а не Go або Java? Які архітектурні переваги це дає?»

Відповідь: Rust забезпечує безпеку пам'яті без збирача сміття (GC), що критично для серверних застосунків із низькою затримкою. На відміну від Go, Rust гарантує відсутність data races на рівні компілятора. На відміну від Java, немає накладних витрат JVM та пауз GC. Для федеративної платформи, де один екземпляр може обробляти тисячі одночасних ActivityPub-запитів, продуктивність та передбачуваність затримок є критичними.

=== Запит 3: Порівняння федеративної та централізованої архітектур соціальних платформ

Назва інструменту та версія: Claude Sonnet 4.6

Мета використання: Формування порівняльного аналізу Lemmy та Reddit за ключовими архітектурними критеріями для слайда та розділу звіту.

Внесок автора: Визначення критеріїв порівняння на основі власного аналізу; самостійний вибір найбільш релевантних пунктів зі запропонованих; написання власного висновку про архітектурні компроміси.

Початковий запит:

«Порівняй федеративну архітектуру Lemmy та централізовану архітектуру Reddit за такими критеріями: точка відмови, масштабування, цензура, відкритість коду, взаємодія з іншими платформами.»

Відповідь: Lemmy: відсутня єдина точка відмови (кожен екземпляр незалежний); горизонтальне масштабування через створення нових екземплярів; цензура лише на рівні окремого екземпляра; повністю відкритий код (AGPL-3.0); взаємодія з усіма платформами ActivityPub (Mastodon, PeerTube). Reddit: єдина точка відмови (збій інфраструктури — недоступність усього сервісу); вертикальне масштабування (один оператор нарощує потужності); централізована модерація та блокування контенту; закритий вихідний код; відсутність інтероперабельності з іншими платформами.
